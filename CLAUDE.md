# CLAUDE.md — постоянные правила проекта

Этот файл Claude Code читает в начале каждой сессии на любой платформе (веб,
десктоп, CLI). Здесь — то, что должно соблюдаться всегда, чтобы контекст не
терялся между сессиями (они на текущем тарифе обрываются резко).

## Про проект
«Реестр патопсихологических методик» — веб-приложение для клинического психолога
(React + TS + SQLite в браузере, локально, без ФИО). Ядро: подбор нормы
Gate → Score. Приложение — поддержка решения специалиста, НЕ диагност (инвариант).
Заказчик — клинический психолог Степан. Код в `app/`.

## В начале каждой сессии
1. Прочитать `clinician-os/handoff.md` и `clinician-os/roadmap.md` — там живой
   контекст, статус и согласованный план.
2. При необходимости — `clinician-os/contour-a-publications.md`,
   `monitoring-and-digest.md`, `shared-base-architecture.md`,
   `norm-selection-and-scoring.md`.

## Дисциплина хендоффа (ГЛАВНОЕ — не терять контекст)
- **После каждого смыслового куска работы** (не только «в конце сессии»)
  обновлять `clinician-os/handoff.md` и `clinician-os/roadmap.md` и **коммитить**.
  Причина: сессии обрываются внезапно — последний коммит хендоффа должен быть
  актуальным в любой момент.
- Любое **решение владельца/клинициста** и любую **развилку** фиксировать в
  документах сразу, со ссылкой, — чтобы не всплывало «мы это обсуждали, но нигде
  не записано».
- Команда `/handoff` выполняет эту процедуру принудительно.

### Защита незавершённой работы от обрыва сессии (3 слоя)
1. **Отметка «в работе» в начале задачи.** Как только берусь за задачу
   разработки — сразу пишу в `handoff.md` блок «В РАБОТЕ: задача, план, какие
   файлы трогаю, критерий готовности» и коммичу. Тогда даже при полной потере
   кода следующая сессия знает, что возобновить и как.
2. **Промежуточные пуши в WIP-ветку.** Незавершённый код часто пушу в
   `claude/gifted-thompson-dsmsv5-wip` (НЕ в рабочую — она автопубликует демо).
   В рабочую `claude/gifted-thompson-dsmsv5` переношу только когда тесты/сборка
   зелёные. Так работа переживает обрыв, а демо остаётся чистым.
3. **Восстановление.** При возобновлении: прочитать handoff/roadmap, `git log`
   и `git status` (включая `…-wip`), определить, что сделано и что потеряно,
   СНАЧАЛА отчитаться пользователю и дождаться подтверждения, и лишь потом
   продолжать. Не коммитить в рабочую ветку сломанное, не принимать решений и
   не открывать PR без явного согласия; на любой развилке — останавливаться и
   спрашивать. Готовый промпт восстановления — в конце `handoff.md`.

## Git и ветка
- Разработка только в ветке `claude/gifted-thompson-dsmsv5`, всегда с пушем
  (демо публикуется автоматически через workflow → GitHub Pages).
- **Автосинхронизация main (решение владельца, июль 2026).** По завершении
  каждой задачи разработки — когда тесты/сборка зелёные и рабочая ветка
  запушена — СРАЗУ подтягивать главную ветку `main` к этой точке
  (`git checkout -B main origin/main && git merge <рабочая> --no-ff && git push origin main`),
  затем вернуться на рабочую ветку. Так main всегда актуален; напоминаний
  владельцу не требуется. main — не источник публикации (сайт идёт с рабочей
  ветки), поэтому его синхронизация безопасна. Разрешение на пуш в main выдано.
- Есть тупиковая ветка `claude/greeting-7rkrnv` (отдельный корень, устаревший
  пустой каркас) — **игнорировать**, в неё не коммитить, за ней не следить.
- PR не открывать без явной просьбы.
- В коммиты/артефакты НЕ писать идентификатор модели.
- Давая пользователю ссылку на файл — давать ПОЛНЫЙ рабочий URL с префиксом
  `clinician-os/…`, а не относительный путь.

## Разрушительные действия (обязательно)
Любое действие, которое **стирает, перезаписывает или необратимо меняет данные
пользователя** (восстановление копии, импорт с перезаписью, удаление, сброс,
замена конфигурации методики), по умолчанию обязано иметь ТРИ вещи:
1. **Подтверждение** — явный шаг «вы уверены», с понятным описанием последствий;
2. **Проверка входа** — убедиться, что данные валидны (тот ли файл/формат), и
   при непригодном входе НИЧЕГО не менять;
3. **Возможность отката** — авто-страховка (напр. скачать текущее перед заменой)
   или версионирование, чтобы ошибку можно было отменить.
И обязательно **тест на «неправильный» путь** (битый/чужой ввод, отмена), а не
только на успешный. Причина: happy-path выглядит рабочим, а беда видна лишь на
краю — именно так в проекте раньше проскочило опасное восстановление копии.
