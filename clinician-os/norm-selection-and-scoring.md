# Раздел ТЗ. Отбор и оценка нормативных источников

> **Назначение раздела.** Определяет, как приложение отбирает, оценивает и выбирает
> нормативные данные (нормы) для сравнения с результатами испытуемого. Раздел
> покрывает: логику отбора нормы под конкретного пациента, скоринговую схему
> качества, пороги, структуру карточки нормы и лог применения.
>
> **Область применения на старте.** Количественные патопсихологические методики
> (таблицы Шульте, «10 слов», счёт по Крепелину). Качественные методики
> подключаются позже — для них действует отдельный каркас (см. раздел 7).

---

## 0. Ключевые принципы (инварианты)

Эти положения обязательны и не подлежат «оптимизации» в реализации.

1. **Приложение — поддержка решения, а не автоматический диагност.** Любой вывод
   об отклонении от нормы подаётся как повод для интерпретации специалистом, а не
   как диагностическое заключение. Финальное подтверждение выбора нормы и
   вердикта — всегда за клиницистом.

2. **База норм — курируемая и версионируемая.** Нормы попадают в production только
   после ручной валидации человеком. Автоматическое извлечение норм-кандидатов
   (краулер / LLM) допустимо только на стадии *предложения*, но не на стадии
   *применения*.

3. **«Лучшая» норма не глобальна — она зависит от пациента.** Норма выбирается для
   пары (норма, конкретный испытуемый), а не назначается один раз «по умолчанию
   для методики».

4. **Отсутствие валидной нормы — легитимный результат.** Если под профиль пациента
   не подходит ни одна норма, система сообщает «валидной нормы нет», а не
   подставляет ближайшую.

5. **Совпадение популяции — не слагаемое, а условие допуска.** Несовпадение
   возраста/образования/языка делает норму невалидной, сколько бы других
   достоинств у неё ни было. Поэтому реализуется двухступенчатая логика:
   **Gate → Score**.

---

## 1. Архитектура отбора: Gate → Score → тай-брейкер

```
Пациент (возраст, образование, язык/культура, [пол])
        │
        ▼
[ Ступень 1: GATE ]  ── отсекает нормы с несовпадающей популяцией
        │
        ▼
Прошедшие кандидаты (возможно, с флагами)
        │
        ▼
[ Ступень 2: SCORE ] ── балл качества 0–100 среди прошедших
        │
        ▼
[ Тай-брейкер ] ── при близких баллах
        │
        ▼
Ранжированный список + рекомендованный дефолт → подтверждение клиницистом
```

Если после Gate список пуст → статус **«норма отсутствует»**, Score не считается.

---

## 2. Ступень 1 — GATE (совпадение популяции)

Вычисляется для каждой пары (норма, текущий пациент). Результат по каждому
критерию: `pass` / `pass_with_flag` / `fail`. Любой `fail` исключает норму из
кандидатов.

| Критерий | pass | pass_with_flag | fail |
|---|---|---|---|
| **Возраст** | пациент внутри возрастной ячейки нормы | пациент на границе ячейки → флаг `edge_of_cell`; **внутри взрослого диапазона 16–60** заход за ячейку до 10 лет → флаг `age_stretch` | пациент вне диапазона нормы; **жёсткие рамки: до 16 и после 60 лет** — отдельные категории, допуск не действует |
| **Образование** | совпадение образовательной страты | норма не стратифицирована по образованию → флаг `no_education_strata` (см. также Score A) | несовпадение страты для методик, где образование критично (счёт по Крепелину; «10 слов» — умеренно) |
| **Язык / материал** | тот же язык предъявления и тот же стимульный материал | невербальная проба (Шульте) на иной культурной выборке → флаг `culture_mismatch` | иной язык на вербальной пробе («10 слов») |

**Примечания к реализации.**
- **Возраст (калибровка клинициста, июль 2026).** До 16 и после 60 лет психические
  функции менее устойчивы — дети и пожилые не входят в общую взрослую норму,
  даже если значения близки. Внутри 16–60 когнитивные нормы «плавающие»:
  интервал 10–15 лет — примерно одна возрастная группа, поэтому заход за ячейку
  до 10 лет допускается с флагом `age_stretch`. Допуск справедлив только для
  когнитивных методик; для личностных отключается (`ageStretchYears: 0`).
- Чувствительность к образованию зависит от методики → порог `fail` по образованию
  конфигурируется на уровне методики (`method_id`).
- Сравнивать пациента следует с нормой **условно здоровых** (`clinical_status`),
  если задача — выявление отклонения. Нормы клинических групп — отдельный сценарий.

## 3. Ступень 2 — SCORE (качество нормы, 0–100)

Считается только для норм, прошедших Gate. Веса ниже — **стартовая калибровочная
точка**, подлежат проверке на реальной базе (см. раздел 6).

### A. Стратификация и размер релевантной ячейки — 30

Оценивается `cell_n` (размер именно той ячейки, куда попал пациент), **не** общий
размер исследования.

| Условие | Баллы |
|---|---|
| `cell_n ≥ 100` и стратификация ≥ по возрасту и образованию | 30 |
| `cell_n` 50–99, стратификация есть | 22 |
| `cell_n` 30–49 | 15 |
| `cell_n` 15–29 | 8 |
| `cell_n < 15` **или** норма дана общим средним без разбивки | 0–4 |

*Обоснование веса:* ширина доверительного интервала вокруг процентиля/среднего в
ячейке напрямую определяет надёжность cut-off для конкретного пациента.

### B. Наличие и форма меры разброса — 25

| Условие | Баллы |
|---|---|
| Полные перцентильные таблицы по ячейкам | 25 |
| Среднее + SD по ячейкам, распределение приемлемо симметрично | 22 |
| Среднее + SD, но распределение скошено (типично для времени) → флаг `skewed_distribution` | 15 |
| Только среднее, без SD и перцентилей | 0–3 |

*Подводный камень:* время выполнения почти всегда скошено вправо и не нормально,
поэтому «среднее ± 2 SD» искажает хвосты. При наличии перцентилей они
предпочтительнее SD — отсюда разница в баллах (см. также раздел 5, расчёт
отклонения).

### C. Идентичность процедуры — 20

Совпадение метрики и протокола с тем, как методику проводит специалист.

| Условие | Баллы |
|---|---|
| Полное совпадение (материал, инструкция, метрика) | 20 |
| Совпадение метрики, мелкие отличия процедуры | 12 |
| Другая метрика / существенно другой протокол → флаг `procedure_mismatch` | 0–5 |

*Обоснование:* «та же методика» по названию ≠ те же числа. Разные источники по
Шульте считают по-разному (сумма времён серии таблиц vs время на одну таблицу).

### D. Свежесть данных — 5 *(итерация 1: было 15)*

По году **сбора** данных (`data_collection_year`); если неизвестен — по
`publication_year` с флагом `year_is_publication`.

| Возраст данных | Баллы |
|---|---|
| ≤ 10 лет | 5 |
| 11–20 лет | 4 |
| 21–35 лет | 2 |
| > 35 лет → флаг `old_data` | 1 |

*Обоснование скромного веса:* дрейф норм реален (эффект Флинна, образовательные
сдвиги), но по русскоязычной патопсихологии старая большая стандартизация нередко
качественнее свежей маленькой. Свежесть не должна перебивать A и B. Старую норму
**не обнуляем** — она может быть единственной валидной.

### E. Формальный статус источника — 20 *(итерация 1: было 10)*

| Условие | Баллы |
|---|---|
| Рецензируемая индексируемая статья **или** официальное руководство по стандартизации | 20 |
| Диссертация / монография / методичка с прописанной процедурой стандартизации | 14 |
| Клинический источник без детального описания стандартизации | 6 |

*Обоснование (итерация 1, по обратной связи клинициста):* статус источника
существенно важнее свежести — вес E поднят с 10 до 20 за счёт снижения D с 15
до 5. **Импакт-фактор как критерий не используется.**

### Итог

```
quality_score = A + B + C + D + E   (0–100)
```

Каждая норма дополнительно несёт список сработавших `flags[]`. Флаги не всегда
снижают балл, но **всегда** отображаются человеку.

## 4. Пороги и статусы

Пороги задают **статус** нормы, а не автопринятие — итоговое слово за клиницистом.

| Балл | `quality_tier` | Поведение системы |
|---|---|---|
| ≥ 75 | **надёжная** | Показывается как рекомендованный дефолт |
| 50–74 | **приемлемая** | Использование допустимо, флаги на виду |
| 30–49 | **слабая** | Только при отсутствии лучшей; крупное предупреждение |
| < 30 | **непригодная** | Хранится (история/справка), в production не предлагается |
| — | **норма отсутствует** | Ни один кандидат не прошёл Gate |

**Тай-брейкер** (разница баллов ≤ 5): сначала выше `A` (стратификация/размер
ячейки), при равенстве — выше `D` (свежесть). Логика: при сопоставимом качестве
берём норму на более точно подходящей и более современной выборке.

## 5. Выбор нормы «в production» — рабочий флоу

1. Клиницист вводит демографию текущего испытуемого (возраст, образование,
   язык/культура; пол — при необходимости).
2. Система прогоняет Gate → Score → тай-брейкер.
3. Система показывает ранжированный список кандидатов и **предлагает дефолт**
   (верхнюю норму), выводя рядом различающие атрибуты каждой: популяция, `cell_n`,
   год сбора, наличие SD/перцентилей, совпадение процедуры, флаги.
4. **Клиницист подтверждает** выбор (может отклониться от дефолта, видя нюансы
   пациента, которых нет в полях: билингвизм, сенсорный дефицит и т.п.).
5. Система рассчитывает отклонение и **логирует применение** (раздел 8).

### Расчёт отклонения (количественные методики)

- **Направление шкалы** берётся из `higher_is_worse` — система не должна путать
  знак отклонения (для времени «больше = хуже», для числа слов «больше = лучше»).
- **Если есть перцентили** — отклонение выражается через перцентиль (предпочтительно
  для скошенных распределений времени).
- **Если только среднее + SD** — z-оценка, но при флаге `skewed_distribution`
  система выводит предупреждение о том, что пороги на хвостах приблизительны.
- Результат подаётся как подсказка («значение соответствует N-му перцентилю /
  отклонение ~z сигм»), а не как категоричный вердикт «патология».

## 6. Калибровка весов (не догма)

**Журнал калибровки.**
- *Итерация 1 (июль 2026, обратная связь клинициста):* D (свежесть) 15 → 5,
  E (статус источника) 10 → 20; возрастной Gate — жёсткие границы 16/60,
  внутри взрослого диапазона допуск захода за ячейку до 10 лет (`age_stretch`).

Веса раздела 3 — гипотеза. После набора реальной базы проверить:

- **Не доминирует ли один критерий.** Если исход почти всегда решает `A` — часть
  его логики, возможно, стоит перенести в Gate.
- **Вменяемо ли число «надёжных» норм** на методику при текущих порогах.
- **Не отсекается ли систематически русскоязычная классика** из-за `D` и `E`.

Схема рассчитана на несколько итераций на живом материале.

## 7. Каркас для качественных методик (отложенный этап)

Числовой скоринг в основном неприменим: «норма» здесь — **типология ответов**, а
не среднее ± SD. Что меняется относительно количественной схемы:

- **Gate по популяции сохраняется** — возраст/образование/язык влияют на
  качественные пробы (пиктограммы, «исключение лишнего») не меньше, часто больше.
- **Объект нормы меняется**: вместо `mean/sd/percentiles` — рубрикатор категорий
  ответов и их квалификация (например, для «исключения лишнего»: обобщение по
  существенному / функциональному / конкретно-ситуативному / латентному признаку —
  и что из этого норма, что снижение обобщения, что искажение).
- **Score смещается** с «размер ячейки + SD» на: качество и воспроизводимость
  критериев квалификации, наличие эталонных примеров, межэкспертную согласованность
  (если сообщается), объём выборки как основание типологии. Критерий B и часть C
  заменяются на «полнота и воспроизводимость рубрикатора».
- **Поведение приложения принципиально иное**: по качественным пробам система
  **не выдаёт** «отклонение N сигм». Максимум — показывает эталонный рубрикатор и
  помогает классифицировать ответ; квалификация остаётся за человеком. Риск ложной
  точности выше, поэтому роль «поддержки, а не диагноста» здесь строже.

> Полную скоринговую схему для качественных методик достраивать на конкретной
> методике с конкретным источником-рубрикатором в руках, иначе получится
> абстракция, требующая переделки.

## 8. Структуры данных

Поля, критичные для Gate и расчёта, помечены `*` — без них норма не может уйти в
production.

### 8.1. Карточка нормы (`Norm`)

**Идентификация**
| Поле | Тип | Назначение |
|---|---|---|
| `norm_id` * | id | Уникальный идентификатор |
| `source_ref` * | text | Библиография / DOI / выходные данные руководства |
| `source_type` * | enum | `indexed_article` \| `standardization_manual` \| `dissertation` \| `monograph` \| `methodical_guide` \| `other` |
| `entered_by`, `entered_at` | user, datetime | Кто и когда внёс |
| `validated_by`, `validated_at` * | user, datetime | Кто провалидировал |
| `validation_status` * | enum | `draft` \| `validated` \| `rejected` \| `archived` |

**Привязка к методике**
| Поле | Тип | Назначение |
|---|---|---|
| `method_id` * | id | Методика (Шульте / «10 слов» / счёт по Крепелину / …) |
| `method_variant` | text | Вариант (какой список слов, серия таблиц, устно/письменно) |
| `measure_type` * | enum | `quantitative` \| `qualitative` (определяет применимость числового скоринга) |
| `metric` * | text | Что измеряется и в каких единицах (сек/таблица; слов на 5-м предъявлении; ошибки; форма кривой) |
| `procedure_notes` | text | Детали проведения для проверки идентичности процедуры |

**Популяция (вход Gate)**
| Поле | Тип | Назначение |
|---|---|---|
| `age_min` *, `age_max` * | int | Границы возрастной ячейки |
| `education_level` * | enum/text | Образовательная страта или «не стратифицировано» |
| `sex` | enum | Если норма его разделяет |
| `language` * | enum | Язык предъявления |
| `culture_region` | text | Популяция/регион |
| `clinical_status` | enum | `healthy` \| `clinical_group` (обычно сравнение с `healthy`) |
| `cell_n` * | int | Размер именно этой ячейки |
| `total_study_n` | int | Общий размер исследования (справочно) |

**Норма (вход расчёта)**
| Поле | Тип | Назначение |
|---|---|---|
| `stat_form` * | enum | `percentile_table` \| `mean_sd` \| `cutoff` \| `qualitative_rubric` |
| `mean`, `sd` | float | Если применимо |
| `percentiles` | map | {перцентиль → значение} |
| `cutoffs` | struct | Готовые пороги (норма / пограничное / патология) |
| `distribution_note` | text | Известная скошенность/форма (флаг для корректного расчёта) |
| `higher_is_worse` * | bool | Направление шкалы (не перепутать знак отклонения) |

**Метаданные качества (вход Score)**
| Поле | Тип | Назначение |
|---|---|---|
| `data_collection_year` | int | Год сбора данных (для свежести) |
| `publication_year` | int | Год публикации |
| `stratified_by` | list | По каким переменным разбита норма |
| `peer_reviewed` | bool | — |
| `quality_score` | int | Рассчитанный балл 0–100 (кэш) |
| `quality_tier` | enum | `reliable` \| `acceptable` \| `weak` \| `unfit` |
| `flags[]` | list | `edge_of_cell`, `skewed_distribution`, `old_data`, `procedure_mismatch`, `no_education_strata`, `culture_mismatch`, `year_is_publication`, … |

**Версионирование / служебное**
| Поле | Тип | Назначение |
|---|---|---|
| `version` * | int | Версия карточки |
| `supersedes` / `superseded_by` | id | Связь версий (старая не удаляется) |
| `active` * | bool | Используется ли сейчас как кандидат |
| `change_log` | list | История правок |
| `applied_count` | int | Сколько раз применялась (аудит/статистика) |

### 8.2. Лог применения (`NormApplication`) — отдельная сущность

Фиксируется при каждом использовании нормы. Делает заключение защитимым: спустя
время видно, какая норма какой версии и почему приложена к конкретному пациенту.

| Поле | Тип | Назначение |
|---|---|---|
| `application_id` | id | Уникальный идентификатор |
| `norm_id` + `version` | ref | Какая норма и какой версии применена |
| `patient_demographics` | struct | Возраст, образование, язык, [пол] на момент применения |
| `raw_value` | float/text | Полученное значение испытуемого |
| `computed_deviation` | struct | Перцентиль / z-оценка + признак скошенности |
| `system_suggestion` | text | Подсказка-вердикт системы |
| `clinician_confirmed` | bool | Подтвердил ли специалист выбор нормы |
| `applied_at`, `applied_by` | datetime, user | Когда и кто применил |

## 9. Чек-лист поведения приложения (сводно)

- [ ] Демография пациента вводится до подбора нормы.
- [ ] Gate отсекает несовпадающие популяции (возраст/образование/язык).
- [ ] При пустом списке — статус «норма отсутствует», а не ближайшая норма.
