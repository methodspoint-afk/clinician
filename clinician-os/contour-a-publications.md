# Контур A — мониторинг публикаций и поиск норм

> **Это не побочная фича, а исходная причина проекта.** Первоначальная
> формулировка Степана (см. `brief from Stepan.md`):
>
> «Я бы хотел накодить приложение, чтобы оно брало все последние исследования
> по патопсихологии и вводило нормы прохождения патопсихологических
> экспериментов в процессе. […] Ты в клинике постоянно одно и то же делаешь —
> после проведения теста постоянно сидишь, ищешь нормы, вводишь».
>
> Контур A — конвейер, который забирает эту рутину: **поиск свежих публикаций →
> извлечение норм → карточка-кандидат → валидация человеком → база**.

## 0. Инварианты (уже зафиксированы в ТЗ, повторяем чтобы не потерять)

1. Автоматическое извлечение норм (ИИ/краулер) — **только стадия предложения**,
   никогда не стадия применения (`norm-selection-and-scoring.md`, §0.2).
2. В production норма попадает **только после ручной валидации** специалистом.
3. О новых исследованиях система **уведомляет**; действующая норма меняется
   **только с согласия** специалиста — автоподмены нет (ТЗ §5).
4. Контур A работает с публикациями, а не с данными пациентов — персональных
   данных в нём нет, на статус «не-медизделия» не влияет.

## 1. Что для Контура A уже построено (ничего не выбрасываем)

| Готово | Где |
|---|---|
| Карточка нормы со всеми полями источника (библиография, тип, годы, ячейка, SD/перцентили, процедура) | `app/src/domain/types.ts` (Norm), форма в `NormsScreen` |
| Автоматическая оценка качества источника (балл 0–100, флаги) | Gate → Score, `norm-selection-and-scoring.md` + код |
| Процесс валидации кандидатов: черновик → валидировать/отклонить, черновики не участвуют в подборе | статусы `draft/validated/rejected/archived` в приложении |
| Версионирование норм (старая не удаляется) | `supersedes`, «Новая версия» в UI |
| Критерии отбора статей | ТЗ §5: аккредитованные журналы по психиатрии/клин.психологии, выборка ≥ 500; качество меряется Score (импакт-фактор не используется) |
| Канал доставки норм всем пользователям (в перспективе) | общая база, `shared-base-architecture.md`, этапы С1–С2 |

**Отсутствует ровно одно звено:** конвейер «поиск публикаций → карточка-кандидат».
Его и строим, поэтапно.

## 2. Этапы

### A0 — ручной ресёрч силами ИИ-ассистента (доступно прямо сейчас, 0 ₽)
- По запросу («сделай ресёрч по Шульте») ассистент ищет свежие публикации
  (eLibrary/КиберЛенинка/Google Scholar/PubMed), скринит по чек-листу (§3)
  и готовит **карточки-кандидаты** в формате приложения.
- Результаты складываются в `clinician-os/research/ГГГГ-ММ-методика.md`
  (лог ресёрча: что искали, что нашли, что отсеяли и почему, кандидаты).
- Степан валидирует кандидатов и вносит через форму (или присылает правки).
- Цель этапа: обкатать чек-лист скрининга и формат карточки-кандидата.

### A1 — регулярный протокол мониторинга
- Фиксированный список источников поиска и поисковых запросов по каждой
  методике (`research/protocol.md`).
- Периодичность: раз в квартал по каждой активной методике (или по запросу).
- Каждый прогон — новый файл лога в `research/`; кандидаты со статусом
  «предложено ресёрчем» и датой прогона.

### A2 — интеграция с приложением (использует формат обмена этапа С0!)
- Импорт карточек-кандидатов файлом → появляются в приложении как «Черновик»
  с пометкой «предложено автоматическим ресёрчем».
- В приложении — уведомление «по методике X появились новые исследования (N)»;
  действующие нормы не трогаются (инвариант 3).

### A3 — серверный конвейер (после С1 из shared-base-architecture.md)
- Кандидаты публикуются в общий каталог как черновики; админы валидируют;
  валидированные нормы получают все пользователи.
- Возможна автоматизация по расписанию (краулер/LLM-агент) — по-прежнему
  только как «предложение».

Связка с общей базой: A2 переиспользует файловый формат С0, A3 — сервер С1.
Контуры не конкурируют, а стыкуются.

## 3. Чек-лист скрининга публикации (вход A0/A1)

Статья проходит в кандидаты, если:
1. Методика совпадает с одной из активных в приложении (та же процедура и
   метрика — иначе `procedure_mismatch`).
2. Указана нормативная выборка: кто (условно здоровые / клиническая группа),
   возрастные ячейки, n по ячейкам (не только общий N).
3. Есть статистика: M±SD и/или перцентили (одно среднее без разброса — слабый
   кандидат, Score сам занизит).
4. Указан язык/культура выборки и год сбора данных (или хотя бы публикации).
5. Тип источника определим (статья/руководство/диссертация/методичка).
Ориентир из ТЗ: выборка исследования от ~500 чел. — желательна, но решает
суммарный Score, а не одно число.

## 4. Формат карточки-кандидата

Совпадает с полями `Norm` (см. `norm-selection-and-scoring.md` §8.1) + два
служебных поля: `candidate_source: research_run_ГГГГ-ММ` и `screening_notes`
(почему прошла чек-лист, что вызывает сомнение). Кандидат всегда рождается со
статусом `draft`.

## 5. Как не потерять это решение (якоря)

- Этот файл — единственный источник правды по Контуру A; на него ссылаются
  `roadmap.md` (этап), `handoff.md` (контекст) и `shared-base-architecture.md`
  (стыковка A2↔С0, A3↔С1).
- Папка `clinician-os/research/` — накопительный журнал всех прогонов ресёрча.
- Любое изменение плана Контура A вносится сюда же, с датой.
