# Общая анонимная база — архитектурный план (дорожка 2)

> **Основание.** Решение клинициста (Степан, июль 2026): «Нужна общая анонимная
> база — это главная фишка проекта. В этом его смысл — сопоставлять нормы и
> организовывать сбор данных». Одновременно: качественные/проективные методики —
> пауза («по ним норм не вывести»), фокус на подсчитываемых.
>
> Статус: проект решения. Код серверной части не пишется, пока владелец не
> подтвердит хостинг и бюджет (§7).

---

## 1. Что строим (одной фразой)

Центральный сервис из двух половин: **(а) общий каталог норм** — курируемый
администраторами, приложения его скачивают; **(б) приём обезличенных
результатов** — приложения отправляют анонимные слепки обследований, из которых
со временем выводятся новые нормы, в т.ч. **по диагнозам** (стандартизация по
патологиям — слова Степана).

Локальное приложение остаётся полностью работоспособным офлайн — сервер его
дополняет, а не заменяет. Это сохраняет главный плюс (лёгкость входа, приватность)
и снижает регуляторные риски.

## 2. Потоки данных

```
Приложение специалиста (локальная SQLite, как сейчас)
   │
   ├── СКАЧИВАЕТ: общий каталог норм (validated, версии, баллы качества)
   │       норм-админы курируют через админ-доступ
   │
   └── ОТПРАВЛЯЕТ (при согласии специалиста, галочка уже есть в UI):
           анонимный слепок результата:
           метод, метрика, значение; возраст, пол, образование;
           диагноз (свободный текст → later МКБ-код); год обследования;
           анонимный код испытуемого (случайный, генерится локально —
           для связки повторных обследований В ПРЕДЕЛАХ одного устройства)
```

**Не покидает устройство никогда:** ФИО (его и нет), локальный код PSY-ГГГГ-XXXX,
комментарии специалиста, препараты, интерпретации, качественные протоколы.

## 3. Роли (по Q14 Степана)

| Роль | Права |
|---|---|
| Специалист | скачивает нормы; отправляет анонимные результаты; всё локальное — как сейчас |
| Администратор норм | + создаёт/правит/валидирует нормы в общем каталоге (только они меняют нормы — прямое требование Степана) |

**Аутентификация без персональных данных (важно для юридики):** вход по
инвайт-коду/токену, который выдаёт владелец. Ни email, ни телефон не собираем →
на старте в системе вообще нет персональных данных пользователей. Анонимные
слепки результатов при правильном обезличивании не являются ПДн. ⚠ Подтвердить
у юриста (152-ФЗ), особенно поле «диагноз» в связке с полом/возрастом/регионом —
при необходимости огрубить (возраст → 5-летние корзины).

## 4. Нормы по диагнозам (флагман — как это вырастает)

1. Слепки копятся в центральной базе с полем «диагноз».
2. Когда ячейка (методика × метрика × возрастная корзина × диагноз) набирает
   n ≥ 30 — система показывает админам **черновик нормы-кандидата** (mean/SD/
   перцентили по накопленным данным + размер ячейки).
3. Админ (Степан) валидирует → норма попадает в общий каталог с
   `clinical_status = clinical_group`, источник = «Общая база проекта, vN».
4. Скоринг Gate → Score работает без изменений — просто появляется новый тип
   источника. Ничто в существующем движке переделывать не надо.

Это и есть «сопоставлять нормы и организовывать сбор данных»: цикл
сбор → агрегация → валидация человеком → применение.

## 5. Технический вариант (рекомендация)

**MVP-сервер:** небольшой REST API (Node/Fastify + PostgreSQL) на российском
VPS/облаке. Почему не BaaS (Supabase/Firebase): хостинг вне РФ, а нам для
российского рынка безопаснее данные в РФ; и функционально нужен очень маленький
API (5–6 эндпоинтов), самописный проще контролировать.

Эндпоинты MVP:
- `GET /norms` — каталог валидированных норм (версии, публичный);
- `POST /submissions` — приём анонимного слепка (токен специалиста);
- `POST/PATCH /norms` — только админ-токены;
- `GET /aggregates` — черновики норм-кандидатов для админов;
- `GET /health`.

В приложении: экран «Синхронизация» (скачать нормы / отправить накопленные
слепки), очередь неотправленного в локальной SQLite, ручной запуск (офлайн-первый
принцип сохраняется).

**Стоимость на старте:** VPS 2 ГБ в РФ ~500–1000 ₽/мес + домен ~600 ₽/год.
Резервные копии — в стоимости VPS (снапшоты) или +150–300 ₽/мес.

## 6. Этапы

| Этап | Содержимое | Зависимости |
|---|---|---|
| С0 (можно сразу, без сервера) | Экран «Синхронизация» в приложении: экспорт слепков в файл + импорт каталога норм из файла. Проверяем формат обмена на Степане до всякого хостинга | нет |
| С1 | REST API + Postgres: каталог норм (read) + приём слепков + админ-валидация | хостинг, домен, токены |
| С2 | Агрегация «нормы-кандидаты по диагнозам» (n ≥ 30), админ-дашборд | наполнение базы |
| С3 | Командный режим (платный — слова Степана), синхронизация устройств | С1 + модель оплаты |

Этап С0 — хитрый ход: формат «слепка» и цикл валидации обкатываем немедленно,
файлами через того же Степана, не тратя ни рубля и не решая юридику. Код С0
на 90 % переиспользуется в С1 (меняется транспорт: файл → HTTP).

## 7. Что нужно от владельца (перед С1)

1. **Бюджет**: ~700–1300 ₽/мес (VPS + домен + бэкапы) — ок?
2. **Хостинг**: российский VPS (рекомендация) — есть ли предпочтения провайдера,
   кто держит аккаунт (владелец продукта, не разработчик)?
3. **Домен**: название (влияет и на бренд).
4. **Администраторы норм**: Степан + кто ещё?
5. **Юрист по 152-ФЗ**: когда планируем консультацию (до публичного запуска С1).

## 8. Что заморожено этим решением

- Качественные/проективные методики: две существующие («Исключение лишнего»,
  «Пиктограммы-протокол») остаются в приложении как есть, но дальше не
  развиваются; новых не добавляем. Норм по ним не будет — слепки по ним в общую
  базу не отправляются.
- Вопросник по качественным методикам (итерация 2) — снят, Степану не отправлять.
