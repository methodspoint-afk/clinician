## 1. Суть в одном абзаце

Приложение — **реестр результатов патопсихологических методик** с динамической базой нормативных данных. Психолог:
1. **Добавляет нормы** из актуальных исследований (источники: Пушкина, MCCB, научные статьи) — это пополняемая, живая база
2. **Кодирует испытуемого** анонимным ID (PSY-YYYY-XXXX, не ФИО)
3. **Вводит время/результаты** теста (время таблиц, ошибки, воспроизведённые слова и т.д.)
4. **Получает автоматическое сравнение** с нормой: результат / норма / отклонение (в σ и процентилях)

**Ключевой эффект**: вместо ручного поиска норм в папках/таблицах — всё тут, быстро, с интерпретацией. По желанию пользователя система может собирать анонимизированные результаты в общую базу для будущей стандартизации и статистики. **Модель многопользовательская** — Owner (Степан) управляет нормами, коллеги/учреждения вводят данные, каждый видит свои результаты.

Ключевой принцип: приложение **предлагает** числа и гипотезы, а итоговый вывод всегда делает врач.

## 2. Пользователи и роли

### MVP состав:
- **Owner (Степан):** администратор системы, управляет нормами, видит всю историю
- **Researchers (коллеги):** психологи, которые проводят обследования, вводят результаты, видят только свои данные
- **[по умолчанию] В MVP:** без Institution-слоя, без многоуровневого управления учреждениями

### Целевая аудитория (fase 2+):
- Клиники и больницы (Institution-уровень с несколькими пользователями)
- Кафедры и обучение студентов
- Частнопрактикующие психологи
- Исследовательские центры, которые хотят вносить свои нормы

### Модель доступа:
- Каждый пользователь видит только **свои** результаты испытуемых (по кодам)
- Owner видит всю базу и может управлять нормами
- **Опциональное участие в общей базе:** пользователь может согласиться отправить свои анонимизированные результаты в общий реестр для стандартизации норм (по возрасту/полу/диагнозу)
- Общая база — это конкурентное преимущество и источник улучшения норм со временем

## 3. Ключевой сценарий (MVP, «счастливый путь»)

### 3.1. Этап подготовки (Owner, один раз)
1. Степан берёт исследование (статья, Пушкина, MCCB) и добавляет нормативные данные:
   - Какая методика, какая возрастная группа, источник с DOI, таблица средних/SD/процентилей
   - Диапазон интерпретации (норма/пограничное/патология)
   - Статус: «в разработке» → «проверена» → «в использовании»
2. База норм пополняется, становится живой, всегда актуальной

### 3.2. Этап обследования (психолог)
1. Создаёт карточку испытуемого: код генерируется автоматически (PSY-2026-0047), вводит только возраст, пол, опционально — диагноз. **Не вводит ФИО.**
2. Выбирает методику из списка (Таблицы Шульте, Коррект. проба, 10 слов, MMPI и т.д.)
3. Вводит «сырые» замеры: время по таблицам, число ошибок, число воспроизведённых слов и т.д.
4. Приложение автоматически:
   - Считает производные показатели по формулам (ЭР, ПУ, ИУ и т.д.)
   - Подбирает подходящую норму (из той, что добавил Owner, по возрастной группе и полу)
   - Если норм несколько — показывает варианты с пояснением (например, «норма по Пушкиной 2015, n=200» vs «норма нашей базы данных, n=450»)
5. Показывает в одной таблице: **Результат | Норма (M±SD) | Отклонение (σ) | Процентиль | Интерпретация**
6. По желанию психолога — добавляет шаблонную осторожную интерпретацию
7. **Опция:** психолог может согласиться отправить этот результат в общую базу (анонимно, код вместо ФИ)
8. Результат сохраняется к карточке испытуемого. При нескольких методиках — доступен сводный отчёт (профиль)

## 4. Методики MVP и логика расчёта

Приложение должно уметь **добавлять новые методики** (кастомизация при покупке лицензии), поэтому методика описывается как «набор вводимых замеров + формулы производных показателей + таблица норм». Ниже — стартовый набор из анкеты.

### 4.1. Заучивание 10 слов
- **Ввод:** число воспроизведённых слов по каждому предъявлению (кривая запоминания), число устойчивых замен слов, отсроченное воспроизведение.
- **Показатели:** кривая запоминания (динамика по предъявлениям), устойчивые замены, отсроченное запоминание.
- **Сравнение с нормой:** по объёму запоминания и форме кривой; отсроченное — отдельно.

### 4.2. Таблицы Шульте (5 таблиц)
- **Ввод:** время заполнения каждой из 5 таблиц (t₁…t₅).
- **Эффективность работы (ЭР):** среднее время таблиц = (t₁+t₂+t₃+t₄+t₅) / 5.
- **Степень врабатываемости (ВР):** t₁ / ЭР.
- **Степень психической устойчивости (ПУ):** t₄ / ЭР.
- **Сравнение с нормой:** по ЭР, ВР, ПУ (с учётом возрастной категории).

### 4.3. Цифровая корректурная проба (адаптация НИИ им. Бехтерева)
- **Ввод:** общее время `tобщ`, время верхней половины `t1`, число ошибок; ошибки отдельно в правой (М1) и левой (М2) половине.
- **Время нижней половины:** `t2 = tобщ − t1`.
  ✓ **Опечатка подтверждена Степаном:** в исходной анкете была ошибка `t2 = t1 − tобщ`, но правильно именно `t2 = tобщ − t1`.
- **Индекс утомляемости (ИУ):** `ИУ = t1 / t2`.
  - ИУ < 1 — повышенная утомляемость;
  - ИУ > 1 — врабатываемость;
  - ИУ = 1 — нормальная психическая активность / общая работоспособность.
- **Коэффициент асимметрии внимания (КАВ):** `КАВ = М1 / М2` (М1 — ошибки справа, М2 — слева). Межполушарное взаимодействие считается нарушенным, если ошибки стабильно преобладают в одной половине зрительного пространства. Норма — **1–2 ошибки**.

### 4.4. Тест зрительно-пространственной памяти
- Оценка **качественная, визуальная**, балл выставляет специалист.
- **[по умолчанию] В MVP:** ручной ввод итогового балла + сравнение с нормой; автоматической обработки изображения нет (Степан отметил, что вживую оценивать проще). Автоматизацию можно проработать позже как отдельную задачу.

## 5. Нормы: справочник и источники

- **Категоризация норм:** норма — это «среднее по популяции ±», но с разбивкой минимум на **дети / взрослые / пожилые**; где есть — по полу и диагнозу.
- **Несколько норм на выбор:** если исследования дают разные цифры, приложение показывает несколько норм с пояснением «для какой категории» (возраст, психиатрический статус) и **всегда предлагает классическую норму на условно здоровой выборке** как опорную. Финальный выбор — за врачом.
- **Источники (из анкеты):**
  - Пушкина Т.П., Пушкина А.В. «Клиническая психология»;
  - MCCB Manual — «MATRICS Consensus Cognitive Battery» (Nuechterlein, Green, UCLA) — там оценки методик разбиты по возрастам в таблицах;
  - актуальные статьи из аккредитованных научных журналов (психиатрия, клиническая психология) с выборкой исследования **не менее 500 человек**.
- **Форма данных в источниках:** в основном таблицы, но часть — картинками (значит, на этапе наполнения базы нужна ручная/ИИ-оцифровка, см. §7).
- **Обновление норм:** новые исследования — только **уведомлять** пользователя; менять действующую норму можно **лишь с его согласия**. Автоматической подмены нормы нет.

## 6. Логика сравнения и вывод результата

- **Базовый вывод:** соотнесение чисел — результат vs норма, направление и величина отклонения.
- **Опциональная интерпретация:** осторожные формулировки уровня «данный параметр занижен, что может указывать на…». Не диагноз, а гипотеза.
- **Межтестовые связи (синдромальная логика):** сопоставление отклонений по разным методикам с типовыми паттернами симптомокомплексов. Пример из анкеты: снижена память при сохранном внимании → скорее невротический регистр, чем общее когнитивное снижение. Опора: Соловьёв С.Л. «Патопсихологические синдромы», пособия Пушкиной.
- **Сводный отчёт (профиль):** по итогам нескольких методик — единое заключение-профиль по испытуемому. Требуется в MVP.
- **История по испытуемому:** несколько тестов в динамике привязаны к карточке.

## 7. Роль ИИ

Два разных контура, их важно не смешивать:

- **Контур A — подготовка базы норм (на этапе разработки/наполнения):** ИИ помогает вытащить нормы из статей и таблиц (в т.ч. с картинок) и привести к единому формату. Это разовая/периодическая работа «за кадром», её результат — выверенная таблица норм. Персональных данных испытуемых здесь нет.
- **Контур B — интерпретация в рантайме (опциональный слой):** ИИ помогает формулировать гипотезы по результату. Обязательные правила:
  - данные во внешний ИИ уходят **только с явного разрешения** пользователя и **только обезличенные**;
  - ИИ **предлагает варианты** (предполагаемый синдром, выраженность отклонений, о чём может говорить), а пользователь **выбирает из вариантов или дополняет своим видением** — готовое заключение «из коробки» не выдаётся;
  - в офлайне показывать предупреждение: *«Вы в офлайн-режиме — нет доступа к интернет-источникам и общей базе; выводы строятся только по ранее загруженным нормам и заключениям».*
- **Возможность жить без рантайм-ИИ:** Степан отметил — когда накопится база обработанных результатов (~500–600), можно отдавать «база + арифметика» и предлагать интерпретации по накопленной статистике, без вызовов внешнего ИИ.

**✓ ПОДТВЕРЖДЕНО СТЕПАНОМ:**
- В **MVP рантайм-ИИ не встраиваем**: норма как выверенная таблица + арифметика + шаблонные осторожные интерпретации. Так проще, дешевле и снимает вопросы врачебной тайны на старте.
- ИИ используем **на этапе разработки** для наполнения базы норм (Контур A) — помощь Степану вытаскивать нормы из статей.
- Когда/если добавляем Контур B (рантайм-интерпретация) — делаем его **опциональным слоем** и **провайдер-независимым** (клиент подключает свой ключ), чтобы данные не проходили через нашу инфраструктуру.
- **Возможность без ИИ:** когда накопится база (~500–600 обследований), приложение может выдавать интерпретации по накопленной статистике, без вызовов внешних сервисов.

## 8. Данные и конфиденциальность

### 8.1. Кодирование испытуемых (критично!)
- **Вместо ФИ:** автогенерируемый код `PSY-YYYY-XXXX` (психолог видит кого тестировал, но код не несет идентифицирующей информации)
- **Почему:** возможность делиться результатами в общую базу без нарушения конфиденциальности
- **Сопоставление:** только сам психолог знает код → испытуемый (может хранить в отдельной закрытой записи)

### 8.2. Хранение данных
- **Локально:** результаты тестов хранятся локально на устройстве психолога (десктоп/мобиль)
- **Облако (опционально):** если психолог согласен — может отправить анонимизированные результаты в общую базу
  - Отправляется: `test_name | age_group | sex | результаты | дата | источник нормы`
  - **НЕ отправляется:** ФИ, диагноз (если есть), личные данные

### 8.3. Внешний ИИ
- Допустимо только с явного разрешения пользователя и только в обезличенном виде
- Данные: коды + возраст/пол + результаты + нормы — всё без ФИ

### 8.4. Общая база (конкурентное преимущество)
- **Цель:** собрать анонимизированные результаты для выведения собственных норм по возрасту/полу/диагнозу
- **Участие:** каждый психолог решает сам, отправлять ли свои результаты (галочка при сохранении результата: «поделиться в общей базе?»)
- **Прозрачность:** психолог видит статистику по общей базе («по 10 словам в возрасте 30-40 лет собрано 340 результатов»)
- Это — основной способ улучшения норм без подписки на новые исследования

## 9. Платформа и технические ориентиры

- **Платформы:** мобильные телефоны **и** десктоп.
- **Интерфейс:** простой, «как базовое приложение из магазина приложений», **не** в стиле 1С/МИС.
- **Офлайн:** желателен. В офлайне — только ранее загруженные нормы и заключения, без современных статей и общей базы; показывать предупреждение (см. §7).
- **[по умолчанию] Техстек:** один кросс-платформенный клиент под мобайл+десктоп из одной кодовой базы (кандидаты — Flutter или веб-приложение/PWA с обёрткой под десктоп). Хранилище локальное; синхронизация/общая база — как более поздний серверный слой. Финальный выбор стека — на этапе промпта на разработку.

## 11. Границы MVP **[СОГЛАСОВАНО С СТЕПАНОМ]**

**Входит в MVP:**
- ✓ Система управления нормами: Owner может добавлять нормы из исследований (таблица: методика, возраст, источник, M±SD, интерпретация)
- ✓ Карточка испытуемого: автокод (PSY-YYYY-XXXX), возраст, пол, диагноз (опц.), локальное хранение
- ✓ 3 методики, доведённые «под ключ»: **Таблицы Шульте**, **Корректурная проба**, **Заучивание 10 слов**
- ✓ Выверенные таблицы норм по возрастным группам (из источников Пушкины, MCCB)
- ✓ Экран результата: `Результат | Норма | Отклонение (σ) | Процентиль | Интерпретация`
- ✓ Сводный отчёт по испытуемому при нескольких методиках
- ✓ Две роли: Owner (управляет нормами) + Researcher (вводит результаты)
- ✓ Опция: согласие на участие в общей базе (галочка при сохранении)
- ✓ Механизм добавления новой методики (конфиг + формулы), заложенный в архитектуру

**НЕ входит в MVP (чтобы не размывать):**
- Рантайм-ИИ и отправка данных во внешние сервисы (только на этапе разработки для наполнения норм)
- Institution-уровень (многоуровневые учреждения с админами)
- Облачная база результатов (каждый видит только свои данные; возможность поделиться — на стадии планирования)
- Автообновление норм из свежих статей
- Автоматическая обработка изображений (зрительно-пространственная память — ручной ввод балла)
- Мобильная версия (MVP — десктоп)

**Критерий успеха MVP:** 
- Степан проводит реальное обследование по 3 методикам, вводит замеры, получает корректный расчёт показателей, сравнение с актуальными нормами и сводный отчёт — **быстрее и без ручного поиска норм**, чем сейчас в Excel/таблицах
- Норм достаточно для работы (реальные таблицы из Пушкины, MCCB)
