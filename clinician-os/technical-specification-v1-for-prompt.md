## 1. Суть продукта в одном абзаце

Приложение — **реестр результатов патопсихологических методик** с курируемой базой
нормативных данных. Клинический психолог вводит демографию испытуемого (под анонимным
кодом, без ФИО) и «сырые» замеры по методике; приложение считает производные
показатели по формулам, **подбирает валидную норму** по двухступенчатой логике
Gate → Score, показывает отклонение (перцентиль / z-оценка) и собирает сводный отчёт
по нескольким методикам. Вместо ручного поиска норм в папках и таблицах — всё в одном
месте, быстро и с прозрачным обоснованием, какая норма и почему применена.

**Главный инвариант:** приложение — поддержка решения, а не автоматический диагност.
Любой вывод — подсказка для интерпретации специалистом; финальное слово всегда за
клиницистом.

---

## 2. Ключевые принципы (обязательны, не подлежат «оптимизации»)

1. **Поддержка решения, а не диагноз.** Отклонение от нормы подаётся как повод для
   интерпретации, не как заключение. Выбор нормы и вердикт подтверждает клиницист.
2. **База норм — курируемая и версионируемая.** Норма попадает в работу только после
   ручной валидации (workflow: черновик → проверена → в использовании). Автоматическое
   извлечение норм (ИИ/краулер) допустимо только как *предложение кандидата*, никогда
   как *применение*.
3. **«Лучшая» норма зависит от пациента.** Норма выбирается для пары
   (норма, конкретный испытуемый), а не назначается один раз для методики.
4. **Отсутствие валидной нормы — легитимный результат.** Если ни одна норма не
   подходит под профиль пациента, система сообщает «валидной нормы нет», а не
   подставляет ближайшую.
5. **Совпадение популяции — условие допуска, а не слагаемое балла.** Несовпадение
   возраста/образования/языка делает норму невалидной независимо от прочих
   достоинств. Отсюда двухступенчатая логика **Gate → Score**.
6. **Конфиденциальность по умолчанию.** ФИО в систему не попадает никогда; данные
   хранятся локально; любая передача наружу — только с явного согласия и только
   в обезличенном виде.

## 3. Пользователи и роли (MVP)

| Роль | Права |
|---|---|
| **Owner** (Степан) | Управляет базой норм (ввод, валидация, версии), управляет справочником методик, видит всю историю применений |
| **Researcher** (коллега-психолог) | Создаёт карточки испытуемых, вводит результаты, видит **только свои** данные |

- **[решение]** В MVP без Institution-слоя (клиники с администраторами — фаза 2+).
- **[решение]** В MVP многопользовательность — **локальные профили на одном
  устройстве** (вход по профилю, данные разделены). Серверная синхронизация и общая
  облачная база — следующий этап; архитектура закладывается так, чтобы серверный слой
  добавлялся без переписывания клиента.

## 4. Основной сценарий («счастливый путь»)

### 4.1. Подготовка базы норм (Owner, периодически)

1. Owner берёт источник (Пушкина, MCCB Manual, статья) и создаёт **карточку нормы**
   по структуре `Norm` (см. раздел 8 и файл раздела о нормах): методика и вариант,
   популяционная ячейка (возраст, образование, язык, пол, клинический статус,
   `cell_n`), статистическая форма (перцентили / M±SD / cut-off), метаданные качества
   (год сбора данных, стратификация, тип источника).
2. Система автоматически считает `quality_score` (0–100) и присваивает
   `quality_tier`; сработавшие флаги сохраняются в карточке.
3. Owner проводит карточку по статусам: `draft` → `validated` → активна.
   Только валидированные и активные нормы участвуют в подборе.
4. Старые нормы не удаляются — версионируются (`supersedes` / `superseded_by`).

### 4.2. Обследование (психолог)

1. Создаёт карточку испытуемого: код генерируется автоматически (`PSY-ГГГГ-XXXX`),
   вводит **возраст, образование, язык, пол; диагноз — опционально. ФИО не вводится.**
2. Выбирает методику из списка.
3. Вводит «сырые» замеры (время таблиц, ошибки, слова по предъявлениям и т.д.).
4. Приложение считает производные показатели по формулам методики (раздел 5).
5. Приложение прогоняет **Gate → Score → тай-брейкер** по базе норм и показывает
   ранжированный список кандидатов с рекомендованным дефолтом. Рядом с каждой
   нормой — различающие атрибуты: популяция, `cell_n`, год сбора, перцентили/SD,
   совпадение процедуры, флаги, `quality_tier`.
   Если Gate не прошла ни одна норма — статус **«валидной нормы нет»**.
6. **Клиницист подтверждает** норму (может выбрать не дефолт — например, из-за
   билингвизма или сенсорного дефицита, которых нет в полях).
7. Экран результата: `Результат | Норма | Отклонение (перцентиль или z) | Статус и флаги | Интерпретация`.
8. По желанию — шаблонная осторожная интерпретация («параметр снижен, что может
   указывать на …» — гипотеза, не диагноз).
9. Каждое применение нормы **логируется** в `NormApplication` (какая норма, какой
   версии, к какой демографии, что насчитано, подтвердил ли клиницист).
10. Результат сохраняется в карточке испытуемого; по нескольким методикам доступен
    **сводный отчёт-профиль**.

## 5. Методики MVP и формулы расчёта

Методика описывается конфигурационно: «набор вводимых замеров + формулы производных
показателей + привязанные нормы + настройки Gate». Добавление новой методики не должно
требовать изменения кода (это же — опция кастомизации при продаже лицензии).

### 5.1. Заучивание 10 слов
- **Ввод:** число воспроизведённых слов по каждому предъявлению, число устойчивых
  замен, отсроченное воспроизведение.
- **Показатели:** кривая запоминания (динамика по предъявлениям), устойчивые замены,
  отсроченное запоминание (сравнивается с нормой отдельно).
- Вербальная методика: язык предъявления в Gate — **критичный** критерий
  (иной язык = `fail`); образование — умеренно критичное.

### 5.2. Таблицы Шульте (5 таблиц)
- **Ввод:** время каждой из 5 таблиц (t₁…t₅).
- **Эффективность работы:** `ЭР = (t₁+t₂+t₃+t₄+t₅) / 5`
- **Степень врабатываемости:** `ВР = t₁ / ЭР`
- **Степень психической устойчивости:** `ПУ = t₄ / ЭР`
- Невербальная проба: иная культурная выборка в Gate — `pass_with_flag`
  (`culture_mismatch`), не `fail`.
- Время выполнения скошено вправо → предпочтительны перцентильные нормы; при
  «M±SD» с флагом `skewed_distribution` — предупреждение (см. 6.2).

### 5.3. Цифровая корректурная проба (адаптация НИИ им. Бехтерева)
- **Ввод:** общее время `tобщ`, время верхней половины `t₁`, число ошибок; ошибки
  в правой (М1) и левой (М2) половинах.
- **Время нижней половины:** `t₂ = tобщ − t₁`
  (✓ подтверждено Степаном; в исходной анкете была опечатка).
- **Индекс утомляемости:** `ИУ = t₁ / t₂`
  - ИУ < 1 — повышенная утомляемость; ИУ > 1 — врабатываемость;
    ИУ = 1 — нормальная психическая активность.
- **Коэффициент асимметрии внимания:** `КАВ = М1 / М2`. Межполушарное взаимодействие
  считается нарушенным, если ошибки стабильно преобладают в одной половине поля.
  Норма — 1–2 ошибки.

### 5.4. Тест зрительно-пространственной памяти — за пределами автоматизации в MVP
- Оценка качественная, визуальная. **[решение]** В MVP — ручной ввод итогового балла
  специалистом и сравнение с нормой, без обработки изображений.

### 5.5. Качественные методики (пиктограммы, исключение лишнего и т.п.) — отложены
- Для них действует отдельный каркас (раздел 7 файла о нормах): Gate по популяции
  сохраняется, но вместо числового отклонения система показывает эталонный рубрикатор
  категорий ответов и помогает классифицировать; квалификация — за человеком.
  В MVP не реализуются, но `measure_type: qualitative` в модели данных заложен.

## 6. Подбор нормы и расчёт отклонения

### 6.1. Подбор: Gate → Score → тай-брейкер

Полная спецификация — в файле `norm-selection-and-scoring.md`. Краткая сводка
для целостности:

- **Gate (допуск).** Для каждой пары (норма, пациент) проверяются возраст,
  образование, язык/материал → `pass` / `pass_with_flag` / `fail`. Любой `fail`
  исключает норму. Чувствительность к образованию **конфигурируется на уровне
  методики**. Сравнение по умолчанию — с нормами условно здоровых
  (`clinical_status = healthy`).
- **Score (качество, 0–100)** считается только для прошедших Gate:
  A. стратификация и размер релевантной ячейки `cell_n` — 30;
  B. наличие и форма меры разброса (перцентили > M±SD) — 25;
  C. идентичность процедуры и метрики — 20;
  D. свежесть данных (по году сбора) — 15;
  E. формальный статус источника — 10. Импакт-фактор не используется.
  Веса — стартовая калибровочная точка, **конфигурируемы** (не хардкод).
- **Пороги:** ≥75 — надёжная (рекомендованный дефолт); 50–74 — приемлемая;
  30–49 — слабая (только при отсутствии лучшей, с крупным предупреждением);
  <30 — непригодная (в подборе не участвует). Пусто после Gate — «норма отсутствует».
- **Тай-брейкер** (разница ≤5 баллов): выше A, при равенстве — выше D.
- Флаги (`edge_of_cell`, `skewed_distribution`, `old_data`, `procedure_mismatch`,
  `no_education_strata`, `culture_mismatch`, `year_is_publication`, …) **всегда
  отображаются** клиницисту.

### 6.2. Расчёт отклонения (количественные методики)

- Направление шкалы — из поля `higher_is_worse` (для времени «больше = хуже», для
  числа слов «больше = лучше»); система не должна путать знак.
- Есть перцентили → отклонение выражается перцентилем (предпочтительно для
  скошенных распределений времени).
- Только M±SD → z-оценка; при флаге `skewed_distribution` — предупреждение, что
  пороги на хвостах приблизительны.
- Подача: «значение соответствует N-му перцентилю / отклонение ~z σ» — подсказка,
  не вердикт «патология».

### 6.3. Межтестовые связи и сводный отчёт

- Сводный отчёт-профиль по итогам нескольких методик — **входит в MVP**: таблица
  всех показателей с отклонениями + текстовая сводка.
- Синдромальная логика (сопоставление паттерна отклонений с типовыми
  симптомокомплексами по Соловьёвой и Пушкиной; пример: память снижена при сохранном
  внимании → скорее невротический регистр) — **[решение]** в MVP как простые
  конфигурируемые правила «паттерн → осторожная подсказка», без ИИ. Полноценный
  синдромальный анализ — фаза 2.

## 7. Роль ИИ (подтверждено Степаном)

- **Контур A — подготовка базы норм (этап разработки/наполнения).** ИИ помогает
  Owner'у вытаскивать нормы из статей и таблиц-картинок и приводить к формату
  карточки `Norm`. Результат всегда проходит ручную валидацию (принцип 2).
  Персональных данных испытуемых в этом контуре нет.
- **Контур B — интерпретация в рантайме.** В **MVP не встраивается**. Когда/если
  добавляется — как опциональный слой, провайдер-независимый (клиент подключает свой
  ключ), данные уходят только с явного разрешения и только обезличенные.
- **Перспектива без ИИ:** при накоплении базы ~500–600 обследований — интерпретации
  по накопленной статистике без внешних вызовов.
- **Офлайн-предупреждение:** «Вы в офлайн-режиме — нет доступа к интернет-источникам
  и общей базе; выводы строятся только по ранее загруженным нормам и заключениям».

## 8. Модель данных (минимальный состав сущностей)

Поля и типы карточки нормы и лога применения — **строго по разделу 8 файла
`norm-selection-and-scoring.md`** (структуры `Norm` и `NormApplication`).
Дополнительно к ним:

### 8.1. `User` — пользователь
`user_id`, `role` (owner | researcher), имя профиля, локальные учётные данные.

### 8.2. `Method` — методика (конфигурируемая)
`method_id`, название, `measure_type` (quantitative | qualitative), список вводимых
замеров (имя, тип, единицы), формулы производных показателей (декларативно, не в
коде), настройки Gate (критичность образования: fail | flag), `higher_is_worse` по
каждому показателю.

### 8.3. `Subject` — испытуемый
`subject_code` (автогенерация `PSY-ГГГГ-XXXX`, уникальный), возраст, **образование
(страта)**, **язык**, пол, диагноз (опционально), `created_by` (владелец-психолог),
дата создания. **Поля ФИО в схеме не существует.**

### 8.4. `TestResult` — результат обследования
`result_id`, `subject_code`, `method_id`, сырые замеры (map), рассчитанные
производные показатели (map), ссылка на `NormApplication` по каждому сравнению,
текст интерпретации (если добавлена), дата, `created_by`, флаг согласия «поделиться
в общей базе» (в MVP только сохраняется как признак).

### 8.5. `Norm` и `NormApplication`
По файлу раздела о нормах, включая версионирование, `validation_status`,
`quality_score`/`quality_tier` (кэш), `flags[]`.

## 9. Данные и конфиденциальность

- **Кодирование:** только автогенерируемый код `PSY-ГГГГ-XXXX`; сопоставление
  код → человек хранит сам психолог вне приложения.
- **Хранение:** локально на устройстве (встроенная БД, напр. SQLite). Экспорт/бэкап —
  локальный файл.
- **Общая база (фаза 2):** отправка анонимизированных результатов
  (`методика | возрастная группа | пол | результаты | дата | норма`) — только по
  явной галочке при сохранении; ФИО и диагноз не отправляются никогда. В MVP —
  только сбор согласий, без сервера.
- **Внешний ИИ:** в MVP из рантайма не вызывается вообще.

## 10. Платформа и техстек

- **MVP — десктоп** (Windows в первую очередь). Мобильная версия — фаза 2, поэтому
  стек должен позволять переиспользование кода.
- **Интерфейс:** простой, уровня «обычное приложение из магазина приложений»;
  не в стиле 1С/МИС. Язык интерфейса — русский.
- **Офлайн-first:** всё ядро (нормы, расчёты, история) работает без сети.
- **[решение] Стек по умолчанию:** кросс-платформенный клиент из одной кодовой базы —
  веб-технологии (TypeScript + React) в обёртке Tauri для десктопа, локальная БД
  SQLite. Альтернатива — Flutter (десктоп + мобайл из одной базы). Финальный выбор
  подтверждается перед стартом разработки; требования к стеку: офлайн-first,
  локальная БД, лёгкая упаковка под Windows, путь к мобильной версии.

## 11. Монетизация и продукт (ориентиры, на архитектуру влияют косвенно)

- Модель — подписка (SaaS) в перспективе; MVP — локальное приложение.
- «Секретный соус» — накопленная база испытуемых, тестов и заключений; она не должна
  утекать (влияет на экспорт и шифрование бэкапов — фаза 2).
- White label нежелателен; контроль и бренд остаются за владельцем.
- Добавление новых методик — опция кастомизации при покупке лицензии (поэтому
  методика конфигурационная, раздел 8.2).

## 12. Границы MVP

**Входит:**
1. Управление нормами по полной схеме `Norm`: ввод, автоматический расчёт
   `quality_score`/`quality_tier`/флагов, валидация, версионирование.
2. Подбор нормы Gate → Score → тай-брейкер с подтверждением клиницистом и статусом
   «валидной нормы нет».
3. Лог применения `NormApplication` по каждому сравнению.
4. Карточка испытуемого с автокодом и демографией (возраст, образование, язык, пол,
   диагноз опц.).
5. Три методики «под ключ»: таблицы Шульте, цифровая корректурная проба, 10 слов —
   с формулами из раздела 5 + ручной балл для зрительно-пространственной памяти.
6. Выверенные таблицы норм из реальных источников (Пушкина, MCCB) — наполняются
   Owner'ом при участии ИИ на этапе подготовки (Контур A).
7. Экран результата и сводный отчёт-профиль; история испытуемого в динамике.
8. Две роли (Owner / Researcher), локальные профили, разделение данных.
9. Механизм добавления новой методики через конфигурацию.
10. Конфигурируемые веса Score и пороги quality_tier.

**Не входит (фаза 2+):**
- Рантайм-ИИ и любые внешние вызовы с данными.
- Institution-слой, серверная синхронизация, общая облачная база (только галочка
  согласия).
- Автообновление норм из свежих статей (только ручное пополнение).
- Автоматическая обработка изображений (зрительно-пространственная память).
- Мобильная версия.
- Качественные методики (каркас в модели данных заложен, реализация отложена).
- Полноценный синдромальный анализ (в MVP — простые правила-подсказки).

## 13. Критерии приёмки MVP

1. Степан проводит реальное обследование по трём методикам, вводит замеры и получает:
   корректные производные показатели, обоснованный подбор нормы (видно, почему эта
   норма и какие у неё флаги), отклонение в перцентилях/σ и сводный отчёт — быстрее,
   чем текущий ручной поиск норм.
2. Для испытуемого, под которого нет валидной нормы, система честно показывает
   «валидной нормы нет» (проверяется тест-кейсом).
3. Каждое применение нормы восстановимо из лога: какая норма, какой версии, к какой
   демографии, что насчитано, подтвердил ли специалист.
4. Направление шкалы не путается: тест-кейсы на «больше = хуже» (время) и
   «больше = лучше» (слова).
5. Норма со статусом `draft` не появляется в подборе.
6. Все расчётные формулы раздела 5 покрыты автотестами с числовыми примерами.
7. Чек-лист раздела 9 файла `norm-selection-and-scoring.md` выполнен полностью.
